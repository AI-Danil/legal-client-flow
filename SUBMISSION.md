# Тестовое задание: Legal Client Flow

## Ссылки

- Рабочий прототип: https://legal-client-flow.netlify.app
- Репозиторий: https://github.com/AI-Danil/legal-client-flow

## Что сделано

Собрал рабочий прототип мини-CRM для юриста: таблица клиентов, добавление клиента с именем/телефоном/статусом, переключение статуса, счетчики по каждому статусу, поиск, фильтр и панель выбранного клиента со следующим шагом. Визуально сделал это как рабочий LegalTech/SaaS-дашборд, а не как учебную таблицу.

Добавил privacy/security слой: данные хранятся только в `sessionStorage`, legacy `localStorage` очищается, телефоны и имена маскируются, PII раскрывается только по кнопке, copy guard блокирует копирование в privacy mode, внешний webhook выключен по умолчанию и требует явного согласия. Для деплоя добавлены security headers: CSP, Referrer-Policy, Permissions-Policy, X-Frame-Options.

Бонус: добавил webhook-настройку. Если указать HTTPS URL из Make, n8n, Zapier или своего backend и включить явное согласие, при добавлении клиента прототип отправляет `POST`. По умолчанию payload маскирует PII; полные имя и телефон отправляются только отдельным чекбоксом.

Бонус 2: Telegram-уведомления без backend. Проверяющий может вставить токен своего тестового бота, нажать «Определить chat ID» (после `/start` боту), отправить тест и получать уведомление о каждом новом клиенте прямо в Telegram. Токен живёт только в sessionStorage вкладки и уходит только в `api.telegram.org`; в интерфейсе явно написано, что использовать нужно тестового бота.

## Лог

Сам сделал продуктовый сценарий, структуру интерфейса, модель данных, логику статусов, privacy/security поведение, проверку сборки и функциональный тест. AI использовал как pair-programmer для ускорения React/TypeScript/CSS-реализации, генерации boilerplate и быстрой проверки UX-формулировок.

Выбрал React + Vite + TypeScript, потому что это самый быстрый путь собрать живой фронтенд-прототип за 2-3 часа и затем без переделки опубликовать его на Vercel/Netlify. Backend не стал добавлять на MVP-этапе: для тестового сценария достаточно session-only хранения, а при развитии данные можно заменить на Supabase/Firebase с авторизацией, RLS и audit log.
