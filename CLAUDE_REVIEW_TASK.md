# ТЗ для Claude Code: review и улучшение Legal Client Flow

Ты работаешь в проекте `legal-client-flow`: React + Vite + TypeScript прототип мини-CRM для юриста.

## Цель

Проведи жёсткий senior-level review продукта, безопасности и реализации. Не переписывай всё с нуля. Сначала найди слабые места, затем предложи точечные улучшения, которые реально усилят тестовое задание для вакансии AI Product Builder.

## Контекст задания

Нужно показать работодателю рабочий прототип:

- юрист видит таблицу клиентов;
- может добавить клиента: имя, телефон, статус дела;
- может менять статус `Новый / В работе / Закрыт`;
- видит счетчики клиентов по статусам;
- бонусом есть автоматизация уведомления через webhook;
- важно показать продуктовое мышление, автономность, скорость и осознанность по данным.

Текущая версия уже содержит:

- React + Vite + TypeScript;
- session-only хранение в `sessionStorage`;
- privacy mode;
- маскирование имени и телефона;
- copy guard;
- явное согласие на webhook;
- security headers для Vercel/Netlify;
- поиск, фильтры, метрики, панель выбранного клиента;
- `README.md`, `SUBMISSION.md`, `vercel.json`, `netlify.toml`.

## Что проверить

1. Product/UX
   - Понятно ли за 5 секунд, что это CRM для юриста?
   - Есть ли реальная польза, кроме CRUD?
   - Насколько убедительно выглядит “следующий шаг”?
   - Не перегружен ли интерфейс для тестового задания?
   - Что добавить за 30-60 минут, чтобы работодатель увидел продуктовый подход?

2. Security/privacy
   - Нет ли ложных обещаний “защиты от копирования”?
   - Корректно ли описано, что frontend не может полностью защитить видимые данные?
   - Нет ли утечек PII в console, URL, localStorage, webhook payload?
   - Нормальный ли CSP для Vercel/Netlify?
   - Есть ли риск из-за разрешённых wildcard webhook domains?
   - Как улучшить privacy mode без backend?

3. Code quality
   - Нет ли TypeScript-проблем, лишней сложности, дублирования?
   - Можно ли упростить state management?
   - Есть ли edge cases: пустые списки, некорректный телефон, сломанный webhook URL, reload/session reset?
   - Нужны ли маленькие утилиты/компоненты вместо большого `App`?

4. Deploy readiness
   - Готов ли проект к Netlify/Vercel?
   - Не мешают ли headers dev-сценарию?
   - Достаточно ли README/SUBMISSION для проверяющего?

## Формат ответа

Сначала дай список находок по severity:

- P0: блокеры;
- P1: важно поправить перед отправкой;
- P2: хорошо бы улучшить;
- P3: nice-to-have.

Для каждой находки укажи:

- файл и участок кода;
- почему это проблема;
- конкретное исправление.

Затем предложи короткий implementation plan на 30-60 минут: максимум 5 пунктов.

Если считаешь нужным вносить код, сделай только точечные изменения с минимальным риском. После изменений обязательно:

```bash
npm run build
```

## Ограничения

- Не добавляй backend, базу данных или auth в этой итерации.
- Не добавляй тяжёлые библиотеки.
- Не ломай текущий визуальный стиль.
- Не делай фейковую безопасность. Лучше честная формулировка и ограниченный, понятный privacy layer.
- Не используй внешние трекеры, CDN, шрифты или аналитики.
